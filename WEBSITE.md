**Two independent outputs from the same source documents:**
1. **RAG pipeline** вҖ” chunks, embeddings, Qdrant, LLM chat (existing, runs on agt.criadoperez.com)
2. **Static website** вҖ” browsable archive of articles, events, books, photo gallery, audio/video links, and chat widget

**Note:** `ficheros/publicos/fotos/` is excluded from the RAG pipeline (`SKIP_FOLDERS` in `convert_documents.py`) вҖ” photos contain no extractable text. They are synced to IPFS normally and scanned directly by `build_catalog.py` for the website catalog.

## Key data flows

### Catalog generation (`build_catalog.py`)

Two-pass catalog builder:

**Pass 1 вҖ” Articles and historical documents** (from Docling output):
1. Walks `output/publicos/` for all `.json` files (Docling output)
2. Reads `origin.filename` from each JSON (first 1KB) to get the original source filename
3. Parses the filename: `YYYY.MMDD.PUBLICATION.TITLE_AUTHOR.ext`
4. Cross-references `ipfs/cids.json` for IPFS download URLs
5. Checks for companion `.md` file (converted article text)

**Pass 2 вҖ” Photos** (directly from source folder):
1. Scans `ficheros/publicos/fotos/` for image files (jpg, jpeg, png, webp, avif, tif)
2. Parses year and caption from filename (best-effort вҖ” see "Photo filename parsing" below)
3. Cross-references `ipfs/cids.json` for IPFS URLs (must run `sync_to_ipfs.py` first)
4. Adds entries with `category: "fotos"` вҖ” no `has_text`, no `publication`

Both passes write to `site/src/data/catalog.json`.

### Photo filename parsing

Photo filenames follow varied, informal conventions. The parser (`parse_photo_filename` in `build_catalog.py`) is best-effort:

```
YYYY.MMDD.CONTEXT.DESCRIPTION.N.ext  вҶ’  date=YYYY-MM-DD, caption from rest
YYYY.MMDD_CODE.ext                   вҶ’  date=YYYY-MM-DD, caption empty (time/WA codes filtered)
YYYY.CONTEXT.N.ext                   вҶ’  date=YYYY,       caption from rest
CONTEXT.ext                          вҶ’  date=None,       caption from whole stem
```

Caption cleaning (`_make_photo_caption`):
- Strips trailing `_AUTHOR NAME` (underscore-prefixed person name after title)
- Strips `_page-NNNN` scan suffixes and `_AGT` author tags
- Splits on dots; filters out pure numbers, `IMG_`/`MG_` camera codes, `WA` codes, timestamps, URLs, resolution strings
- Strips trailing series numbers (`.1`, `.2`, `(1)` etc.) from the last part
- Applies title case to all-uppercase or all-lowercase parts; leaves mixed-case as-is
- Joins remaining parts with ` В· `

Known limitations:
- Facebook/social media photo IDs (`1070086_398515276924259_n.jpg`) produce no meaningful caption
- Filenames with year embedded mid-string (`LIBRERIA RUEDO IBERICO.PARIS.1970.jpg`) are placed in "Sin fecha" since there is no year prefix
- Informal URLs in filenames (`1972. httpsfotos.europapress...`) partially leak into the caption

### Filename convention

```
YYYY.MMDD.PUBLICATION.TITLE_AUTHOR.ext

Examples:
  1977.0524.REPORTER.01.REFORMA Y REPRESIГ“N_AGT.docx
  2003.0815.LA RAZON.ANARQUISMO_AGT.docx
  1968.0515.INFORME A CARRERO BLANCO SOBRE GUINEA_AGT.docx  (no publication)
  1976.0000.LA LEY DE PRENSA_AGT...docx                     (month unknown)

Known edge cases:
  - 1994-0117  (dash instead of dot вҖ” typo)
  - 2002.0'318 (stray apostrophe вҖ” typo)
  - 2004.0105, (comma instead of dot вҖ” typo)
  - REPORTER.01.TITLE (numbered series)
```

### Deduplication

The `articulos/` folder contains files at root level AND organized copies in `DOC/`, `JPG/`, `PDF/` subfolders. The Docling conversion pipeline produces separate outputs for each. The catalog builder deduplicates by `origin.filename` вҖ” it keeps the shallowest-path entry (root-level preferred over subfolder copies). DOC/JPG/PDF subfolder entries are deprioritized with a depth penalty of +100.

Result: 3,232 Docling outputs вҶ’ 1,944 unique catalog entries after dedup.

### HECHOS subcategories

HECHOS events are identified by the first-level subfolder under `AGT.HECHOS/`. Deeper nesting (e.g., `1971-DIARIO MADRID/DIARIO MADRID/subfile.pdf`) is flattened into the top-level event. This is handled in `build_catalog.py` line ~330.

### Article page rendering

Astro reads `catalog.json` at build time. For each article with `has_text: true`, it reads the corresponding `.md` file from `output/publicos/`, renders it to HTML with `marked`, and generates a static page.

### Chat widget

The chat page embeds a JavaScript widget that calls the RAG API:

- **Endpoint:** `POST https://agt.criadoperez.com/chat`
- **Request:** `{ messages: [{ role: "user", content: "question" }] }`
- **Response:** SSE stream with events:
  - `{ type: "sources", chunks: [...] }` вҖ” retrieved document chunks
  - `{ type: "text", text: "..." }` вҖ” streamed LLM tokens
  - `{ type: "done" }` вҖ” end of response

## Adding new content

### Articles / historical documents
1. Drop new files in `ficheros/publicos/` (follow the naming convention)
2. Run the pipeline:
   ```bash
   python convert_documents.py      # Convert to markdown
   python chunk_documents.py        # Chunk for RAG
   python sync_to_ipfs.py           # Pin to KUBO
   python embed_and_index.py        # Update vector DB
   python build_catalog.py          # Regenerate website catalog
   cd site && npm run build         # Rebuild static site
   ```

### Photos
Photos go in `ficheros/publicos/fotos/`. They are **excluded from the RAG pipeline** (`SKIP_FOLDERS` in `convert_documents.py`) вҖ” only IPFS sync and the catalog builder run on them:
```bash
python sync_to_ipfs.py       # Pin photos to KUBO (adds CIDs to ipfs/cids.json)
python build_catalog.py      # Scan fotos/ and add to catalog
cd site && npm run build     # Rebuild static site
```
No `convert_documents.py`, `chunk_documents.py`, or `embed_and_index.py` needed.

New text categories: create a new subfolder under `ficheros/publicos/`. The catalog builder auto-discovers all subdirectories. Add a corresponding Astro route in `site/src/pages/`.
New non-text categories (like `fotos`): add the folder name to `SKIP_FOLDERS` in `convert_documents.py`, then add scanning logic to `build_catalog.py` and an Astro route.

## Known limitations

- **42 HECHOS documents have no parsed date** вҖ” their filenames don't follow the `YYYY.MMDD` convention (e.g., `0.CONSTITUCION DE GUINEA.1973.W.pdf`, `CITAS DE AGT SOBRE GUINEA.docx`). They still appear in the site with their title but no date.
- **Photos require IPFS sync before catalog build** вҖ” if `sync_to_ipfs.py` hasn't run for a photo, it has no `ipfs_url` and is silently skipped by the gallery page. Always run `sync_to_ipfs.py` before `build_catalog.py` when adding new photos.
- **Photo caption quality** вҖ” captions come entirely from filename parsing (no manual metadata). Filenames with Facebook/social IDs, YouTube links, or non-standard formats produce empty or noisy captions. See "Photo filename parsing" section for details.
- **fotos/ excluded from RAG** вҖ” `SKIP_FOLDERS = {"fotos"}` in `convert_documents.py`. Intentional: photos yield no useful text for the vector DB. Add other non-text folder names to `SKIP_FOLDERS` as needed.
- **Slug generation** in `hechos/index.astro` and `hechos/[event].astro` must stay in sync вҖ” both compute slugs from subcategory folder names using the same regex. If one changes, links will break.
- **New publications** not in the `KNOWN_PUBLICATIONS` set in `build_catalog.py` will not be separated from the title. Add them to the set when new publications appear.
- **Pagefind** runs as a post-build step. The `npm run build` script chains it: `astro build && npx pagefind --site dist`.
- **Chat widget** hardcodes the API URL to `https://agt.criadoperez.com/chat`. Change it in `site/src/pages/chat.astro` if the API moves.
