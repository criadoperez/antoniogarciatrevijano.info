# Website: www.antoniogarciatrevijano.info

Memorial and research archive for Antonio García-Trevijano. Static site built with Astro, hosted on IPFS.

## Build status

Phase 1 complete. The site generates 1,887 static pages:
- 1,870 individual article pages with full text and IPFS download links
- 10 historical event pages (73 documents total) with collapsible text
- 11 books listed (no download — private)
- Landing page, audio/video links (iVoox/YouTube), AI chat page
- Client-side full-text search via Pagefind (1,856 pages indexed, 50,543 words)
- Prev/next navigation on every article page

```bash
# Quick build
cd site && npm run build

# Full rebuild (catalog + site + search index)
cd /root/agt && ./build.sh
```

## Architecture overview

```
ficheros/publicos/           Source documents (articles, historical docs)
        │
        ├── convert_documents.py  →  output/publicos/*.md *.json   (existing RAG pipeline)
        ├── chunk_documents.py    →  chunks/chunks.jsonl
        ├── sync_to_storacha.py   →  storacha/cids.json            (IPFS upload)
        ├── embed_and_index.py    →  qdrant_db/                    (vector DB)
        │
        └── build_catalog.py      →  site/src/data/catalog.json    (NEW: website catalog)
                                          │
                                     astro build
                                          │
                                     site/dist/   →  IPFS / web server
```

**Two independent outputs from the same source documents:**
1. **RAG pipeline** — chunks, embeddings, Qdrant, LLM chat (existing, runs on agt.criadoperez.com)
2. **Static website** — browsable archive of articles, events, books, audio/video links, and chat widget

## Key data flows

### Catalog generation (`build_catalog.py`)

1. Walks `output/publicos/` for all `.json` files (Docling output)
2. Reads `origin.filename` from each JSON (first 1KB) to get the original source filename
3. Parses the filename: `YYYY.MMDD.PUBLICATION.TITLE_AUTHOR.ext`
4. Cross-references `storacha/cids.json` for IPFS download URLs
5. Checks for companion `.md` file (converted article text)
6. Outputs `site/src/data/catalog.json`

### Filename convention

```
YYYY.MMDD.PUBLICATION.TITLE_AUTHOR.ext

Examples:
  1977.0524.REPORTER.01.REFORMA Y REPRESIÓN_AGT.docx
  2003.0815.LA RAZON.ANARQUISMO_AGT.docx
  1968.0515.INFORME A CARRERO BLANCO SOBRE GUINEA_AGT.docx  (no publication)
  1976.0000.LA LEY DE PRENSA_AGT...docx                     (month unknown)

Known edge cases:
  - 1994-0117  (dash instead of dot — typo)
  - 2002.0'318 (stray apostrophe — typo)
  - 2004.0105, (comma instead of dot — typo)
  - REPORTER.01.TITLE (numbered series)
```

### Deduplication

The `articulos/` folder contains files at root level AND organized copies in `DOC/`, `JPG/`, `PDF/` subfolders. The Docling conversion pipeline produces separate outputs for each. The catalog builder deduplicates by `origin.filename` — it keeps the shallowest-path entry (root-level preferred over subfolder copies). DOC/JPG/PDF subfolder entries are deprioritized with a depth penalty of +100.

Result: 3,232 Docling outputs → 1,944 unique catalog entries after dedup.

### HECHOS subcategories

HECHOS events are identified by the first-level subfolder under `AGT.HECHOS/`. Deeper nesting (e.g., `1971-DIARIO MADRID/DIARIO MADRID/subfile.pdf`) is flattened into the top-level event. This is handled in `build_catalog.py` line ~330.

### Article page rendering

Astro reads `catalog.json` at build time. For each article with `has_text: true`, it reads the corresponding `.md` file from `output/publicos/`, renders it to HTML with `marked`, and generates a static page.

### Chat widget

The chat page embeds a JavaScript widget that calls the RAG API:

- **Endpoint:** `POST https://agt.criadoperez.com/chat`
- **Request:** `{ messages: [{ role: "user", content: "question" }] }`
- **Response:** SSE stream with events:
  - `{ type: "sources", chunks: [...] }` — retrieved document chunks
  - `{ type: "text", text: "..." }` — streamed LLM tokens
  - `{ type: "done" }` — end of response

## Adding new content

1. Drop new files in `ficheros/publicos/` (follow the naming convention)
2. Run the pipeline:
   ```bash
   python convert_documents.py      # Convert to markdown
   python chunk_documents.py        # Chunk for RAG
   python sync_to_storacha.py       # Upload to IPFS
   python embed_and_index.py        # Update vector DB
   python build_catalog.py          # Regenerate website catalog
   cd site && npm run build         # Rebuild static site
   ```
3. Deploy `site/dist/` to IPFS or web server

New categories: just create a new subfolder under `ficheros/publicos/`. The catalog builder auto-discovers all subdirectories. Add a corresponding Astro route in `site/src/pages/`.

## Known limitations

- **42 HECHOS documents have no parsed date** — their filenames don't follow the `YYYY.MMDD` convention (e.g., `0.CONSTITUCION DE GUINEA.1973.W.pdf`, `CITAS DE AGT SOBRE GUINEA.docx`). They still appear in the site with their title but no date.
- **Slug generation** in `hechos/index.astro` and `hechos/[event].astro` must stay in sync — both compute slugs from subcategory folder names using the same regex. If one changes, links will break.
- **New publications** not in the `KNOWN_PUBLICATIONS` set in `build_catalog.py` will not be separated from the title. Add them to the set when new publications appear.
- **Pagefind** runs as a post-build step. The `npm run build` script chains it: `astro build && npx pagefind --site dist`.
- **Chat widget** hardcodes the API URL to `https://agt.criadoperez.com/chat`. Change it in `site/src/pages/chat.astro` if the API moves.
